# Token Handoff — orchard-ui client-side exception on `/groves/[id]`

## Task
User reported: browsing to `http://localhost/groves/297fe996-0807-47ca-90d8-83246e575f42`
shows Next.js's generic `Application error: a client-side exception has occurred`.
Goal: find and fix the root cause (systematic-debugging skill in progress, Phase 1 —
root cause not yet fully confirmed).

## Confirmed root cause chain (reproduced, not yet fixed)

1. Reproduced with a headless-Chromium Playwright script hitting
   `http://localhost:80/groves/297fe996-0807-47ca-90d8-83246e575f42/` (script saved at
   `/tmp/claude-1000/-home-steve-code-orchard-cde-orchard-ui/ab468afb-1679-4f32-88b0-8a998d0a5717/scratchpad/reproduce80.js`,
   Playwright + Chromium already installed to `~/.cache/ms-playwright/chromium-1228`).
2. Real browser error captured:
   ```
   ChunkLoadError: Loading chunk 102 failed.
   (error: http://localhost/_next/static/chunks/app/(main)/groves/%5Bid%5D/page-fa1387a45ed02bad.js)
   ```
   That JS chunk request comes back **HTTP 400** with a Tomcat-style default error
   page body (`HTTP Status 400 – Bad Request`, the grey-header Tomcat error.jsp style) —
   so it's an upstream Java/Spring Boot (embedded Tomcat) rejection, not a browser/JS bug.
3. **This 400 only happens when the request goes through nginx on port 80.** The exact
   same URL, hit directly against port 3000 (bypassing nginx), returns **200** with the
   correct JS content (`Content-Type: text/javascript`, `Last-Modified: Mon, 29 Jun 2026`).
   Reproduced this discrepancy twice (not a one-off race).
4. User confirmed **port 3000 is the actual Canopy UI Java app** (Spring Boot, embedded
   Tomcat, serving the prebuilt static Next.js export via `SpaResourceConfig` — see
   `backend/src/main/java/dev/orchard/ui/backend/web/SpaResourceConfig.java`). This
   contradicts `backend/src/main/resources/application.yml`'s `server.port: 8080` default,
   so this instance must have been started with a port override — worth checking how/where
   (env var `SERVER_PORT=3000`, `--server.port=3000` arg, etc.) if it matters for the fix.
5. nginx config at `/etc/nginx/sites-enabled/orchard-proxy.conf`
   (symlink → `/etc/nginx/sites-available/orchard-proxy.conf`):
   ```nginx
   server {
       listen 80;
       server_name _;

       location /api/ {
           proxy_pass http://127.0.0.1:8080;
           proxy_set_header Host $host;
           proxy_set_header X-Real-IP $remote_addr;
           proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
           proxy_set_header X-Forwarded-Proto $scheme;
       }
       location / {
           proxy_pass http://127.0.0.1:3000/;
           proxy_set_header Host $host;
           proxy_set_header X-Real-IP $remote_addr;
           proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
           proxy_set_header X-Forwarded-Proto $scheme;
       }
   }
   ```
   `/etc/nginx/error.log` was 0 bytes when checked — nginx is not logging an nginx-level
   error for this request, consistent with it being a pass-through of the upstream's 400.

## Leading hypothesis (NOT yet verified — this is where to pick up)

`location /` matches the whole path and `proxy_pass http://127.0.0.1:3000/;` has a
trailing `/`. That combination makes nginx decode/reassemble the request URI before
proxying (rather than passing the raw request line through verbatim), which is the
classic nginx gotcha for URLs containing percent-encoded reserved characters
(`%5B`/`%5D` here) mixed with literal reserved characters (`(`/`)`). The theory is nginx
re-encodes or decodes something in `.../app/(main)/groves/%5Bid%5D/page-*.js` in a way
that Tomcat's strict URI character validation then rejects, even though the same literal
request succeeds when sent directly.

**Not yet confirmed.** Next steps to actually nail it down:
- Capture the raw bytes of the request nginx sends upstream — easiest way: temporarily
  swap `proxy_pass` to point at a `nc -l -p <port>` listener (or use `tcpdump -A -i lo
  port 3000`) and diff the request line against the direct-curl version.
- Test whether removing the trailing slash from `proxy_pass http://127.0.0.1:3000;`
  (no `/`) changes the behavior — that changes nginx from "replace matched prefix" mode
  to "pass full original URI unmodified" mode, which may sidestep the re-encoding.
- Once confirmed, the fix is an nginx config change, not application code. Check whether
  `/etc/nginx/sites-available/orchard-proxy.conf` is tracked anywhere in the repo (e.g.
  `orchard-cde/infra` or `orchard-cde/operations`) so the fix lands as a tracked change
  rather than a hand-edit on the box — I did not check this yet.
